株式会社ZENSHINの高橋です。

2026 年 4 月 1 日に Cloudflare が EmDash というヘッドレス CMS の `v0.1.0 preview` を出しました。WordPress の後継を目指す触れ込みで、Astro の上に管理画面を乗せた構成です。

[https://blog.cloudflare.com/emdash-wordpress/](https://blog.cloudflare.com/emdash-wordpress/)

ブラウザだけで試せる Playground もあり、Cloudflare アカウントなしですぐに試せます。

[https://emdashcms.com/](https://emdashcms.com/)

最近、WordPress でのホームページ制作について相談を受けることがあります。自分たちのサイトも Cloudflare 寄りの構成で運用していることもあり、Cloudflare から出てきた EmDash が気になりました。そこで、Playground を試し、公式ドキュメントを読んだうえで、WordPress / Astro と比べながら所感を書いておきます。

## 結論: 更新者で選ぶ

3 つを見比べた結果、機能差の優劣よりも **「日常的に誰が更新するか」** と **「非エンジニアにどこまで触ってほしいか」** で決まる、という整理に落ち着きました。

| 観点                   | WordPress                       | EmDash                           | Astro                              |
| ---------------------- | ------------------------------- | -------------------------------- | ---------------------------------- |
| 主な更新者             | 非エンジニア                    | 非エンジニア + エンジニア        | エンジニアだけ                     |
| 非エンジニアが触る範囲 | 見た目 + 記事（管理画面で完結） | 記事だけ（見た目はコードで管理） | （触らない / PR レビューで関わる） |

エンジニアだけで更新するなら、CMS を挟まず Astro で Markdown を書くほうが軽いです。非エンジニアに見た目まで任せたい案件では、WordPress のページビルダー文化がいまも根強く残っています。EmDash はちょうどその中間で、**「記事は管理画面、見た目はコード」という分担で動きます**。

以下、その判断につながった差分を 3 つの観点で見ていきます。

## 更新フローとコード管理

「誰が更新するか」の境界は、「Git のコードでどこまで持つか」の境界と表裏一体です。

| 観点                 | WordPress                           | EmDash                   | Astro                      |
| -------------------- | ----------------------------------- | ------------------------ | -------------------------- |
| 記事・コンテンツ     | 管理画面（DB）                      | 管理画面（D1）           | マークダウンファイル = Git |
| スキーマ定義         | 管理画面                            | 管理画面                 | `content.config.ts` = Git  |
| デザイン・レイアウト | 管理画面（テーマ + ページビルダー） | Astro プロジェクト = Git | Git                        |

面白いと思ったのは、コレクションやフィールドの追加（スキーマ定義）を管理画面から直接できる点です。Astro の Content Collections だと `content.config.ts` を書き換えて Git push が要るので、項目をひとつ増やすだけでもデプロイが走ります。EmDash では、その作業が管理画面だけで完結します。

## ホスティング先とコンテンツの持ち方

デプロイ先とコンテンツの持ち方にも、かなり差があります。

| 観点               | WordPress                                     | EmDash                                               | Astro                        |
| ------------------ | --------------------------------------------- | ---------------------------------------------------- | ---------------------------- |
| ホスティング先     | レンタルサーバー / VPS / マネージド WordPress | Cloudflare Workers（推奨） / Netlify / Vercel / Node | 任意（静的ホスティングで可） |
| コンテンツの保存先 | MySQL                                         | Cloudflare D1（ローカル開発時は SQLite）             | リポジトリ内のファイル       |
| データ形式         | HTML 混じりの投稿本文                         | Portable Text（構造化 JSON）                         | マークダウン                 |

EmDash は Cloudflare 製で、Cloudflare Workers + D1 の組み合わせが一番素直に動きます。Netlify / Vercel / Node サーバーでも動きますが、D1 を前提にするならやはり Workers です。コンテンツは Portable Text（構造化 JSON）で持ちます。感触は Sanity に近く、マークダウンをそのまま扱う Astro Content Collections とはかなり違います。

## セキュリティと運用の構造

| 観点           | WordPress                            | EmDash                                     | Astro                  |
| -------------- | ------------------------------------ | ------------------------------------------ | ---------------------- |
| プラグイン実行 | 同一 PHP プロセス内                  | Dynamic Worker の isolate でサンドボックス | （プラグイン機構なし） |
| 配信形態       | 動的レンダリング（キャッシュで緩和） | 静的 + エッジ                              | 静的（ビルド時生成）   |
| 高負荷対策     | キャッシュ層を重ねる                 | Cloudflare エッジに乗る                    | CDN 配信で素直に速い   |

WordPress のプラグインが同一プロセスで動くモデルは、運用する中で気を使う場面が多くありました。EmDash がプラグインを isolate に閉じ込めているのは、WordPress で改善しにくかった部分にうまく手当てしていると感じます。

## 更新体制と前提条件で選ぶ

- 非エンジニアに見た目まで触ってほしいなら、WordPress が選びやすいです
- 記事は非エンジニアが更新し、見た目はエンジニアがコードで管理するなら、EmDash は選択肢に入ります
- Cloudflare 基盤に寄せていて、D1 / Workers 前提の構成と相性がよい場合も、EmDash は検討しやすいです
- エンジニアだけで更新するなら、Astro のようにコンテンツも Git で持つ構成のほうがシンプルです
- EmDash を選ぶなら、現時点では `v0.1.0 preview` であることや、プラグインマーケットプレイス・WooCommerce 相当の EC・日本語情報がまだ限られている点は前提になります

## 現時点での判断

このホームページ自体は Astro で動かしていて、更新はエンジニアが PR でやっています。エンジニアだけで更新する前提なら、いまのところこの構成が比較的軽く運用できています。

[https://contents.zenshin-inc.co.jp/blog/tech-selection/](https://contents.zenshin-inc.co.jp/blog/tech-selection/)

現時点では、WordPress ほど管理画面に寄っているわけでもなく、Astro ほどシンプルでもないので、正直まだ中途半端な印象もあります。

WordPress からの移行先として考えるなら、管理画面でできることがもう少し増えて、非エンジニアでも完結しやすい方向に寄っていく必要がありそうです。逆に、開発者向けの使い勝手をもっと磨く方向に振れるなら、それはそれで別の立ち位置が見えてきそうです。しばらくは、その振れ方を見ていこうと思います。
